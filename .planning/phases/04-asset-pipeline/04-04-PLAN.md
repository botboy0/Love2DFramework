  <name>Task 1: Stub print in harness spec and suppress Bus log in main_spec</name>
  <files>tests/helpers/plugin_harness_spec.lua, tests/main_spec.lua</files>
  <action>
Fix two test files that leak output to stdout during busted runs:

1. **tests/helpers/plugin_harness_spec.lua** (line ~55, tolerant-mode test):
   The test at line 55 ("does not error in tolerant mode for undeclared service") calls `ctx.services:get("crafting")` which triggers `print("[Harness] ...")` inside `make_dep_enforced_services` (plugin_harness.lua line 50).

   Fix: Wrap the test body with a print stub. Before the `ctx.services:get("crafting")` call, save `_G.print` to a local, replace `_G.print` with `function() end`, and restore it in a `finally` block (or just after the assertion). Example pattern:
   ```lua
   it("does not error in tolerant mode for undeclared service", function()
       local crafting_svc = { _name = "crafting" }
       local ctx = harness.create_context({
           deps = { crafting = crafting_svc },
           allowed_deps = {},
           error_mode = "tolerant",
       })
       local original_print = _G.print
       _G.print = function() end
       -- In tolerant mode, accessing an undeclared service warns but does not error
       assert.has_no_error(function()
           ctx.services:get("crafting")
       end)
       _G.print = original_print
   end)
   ```

2. **tests/main_spec.lua** (line ~309, Bus.new defaults to tolerant test):
   Line 309 creates `Bus.new({ error_mode = bus_error_mode })` without a log suppressor. When the tolerant-mode bus flushes a handler that errors, Bus logs the error via its log function — which defaults to `print` if none provided.

   Fix: Pass a no-op log function to suppress the output:
   ```lua
   local bus = Bus.new({ error_mode = bus_error_mode, log = function() end })
   ```
   This changes only line 309. The test still validates that tolerant mode does not propagate errors — it just stops the log output from leaking to stdout.

<task type="auto">
  <name>Task 2: Guard print calls behind opts.silent in validate_architecture.lua</name>
  <files>scripts/validate_architecture.lua</files>
  <action>
The `Validator.run()` function accepts `opts.silent` (documented at line 667) but never checks it. All `print_section`, `print_warning_section`, and inline `print` calls execute unconditionally. This causes "Fixed: created 1 missing test file(s)" to appear in test output when specs call `Validator.run({ fix = true, silent = true })`.

Fix: Thread `opts.silent` through to suppress all print output when true.

Specific changes in `Validator.run()`:

1. After `local verbose = opts.verbose or false` (line 672), add:
   ```lua
   local silent = opts.silent or false
   ```

2. Create a conditional print helper at the top of `Validator.run()`:
   ```lua
   local function log(...)
       if not silent then
           print(...)
       end
   end
   ```

3. Replace ALL `print(` calls inside `Validator.run()` with `log(`:
   - Line 770: `print("Fixed: created " .. fixed .. " missing test file(s)")` -> `log(...)`

4. Modify `print_section` and `print_warning_section` to accept a `silent` parameter. The simplest approach: add an early return at the top of each function if a new `silent` parameter is true. Then pass `silent` from all call sites in `Validator.run()`.

   Alternative (simpler): Replace the module-level `print_section` and `print_warning_section` calls in `Validator.run()` with a pattern that checks `silent` before calling them:
   ```lua
   if not silent then
       print_section(...)
   end
   ```
   Apply this guard to ALL six `print_section`/`print_warning_section` calls in `Validator.run()` (lines 704, 728, 752, 776, 814, 817, 860, 863).

5. Guard the script entry point prints too (lines 893-903) — these use `opts` from arg parsing. Add `silent = false` to the opts table passed to `Validator.run()` at the script entry point (this is already the default, but be explicit). The entry-point prints after `Validator.run()` returns should NOT be silenced (they are CLI output, not library output). Only `Validator.run()` internal prints need silencing.

Do NOT change the function signatures of `Validator.detect_*` methods — they are pure and do not print.
Do NOT change the script entry point's own print statements (lines 893-903) — those are correct CLI output.
