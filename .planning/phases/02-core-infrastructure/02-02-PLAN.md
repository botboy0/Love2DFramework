**src/core/worlds.lua** — Dual world factory:

evolved.lua is a **global singleton** ECS — it does not support multiple "world" instances via constructor. Instead, the dual-world separation must be achieved through **namespace isolation**: both server and client use the same evolved module, but tag their entities with a `ServerEntity` or `ClientEntity` fragment. Queries for server systems include `ServerEntity`, client systems include `ClientEntity`.

```lua
local evolved = require("lib.evolved")
local Components = require("src.core.components")

local Worlds = {}

-- Create fragments to tag which "world" an entity belongs to
Worlds.ServerTag, Worlds.ClientTag = evolved.id(2)

function Worlds.create()
    return {
        server = { tag = Worlds.ServerTag },
        client = { tag = Worlds.ClientTag },
        -- Helper: spawn entity in a specific world
        spawn_server = function(self, components)
            components = components or {}
            components[Worlds.ServerTag] = true
            return evolved.spawn(components)
        end,
        spawn_client = function(self, components)
            components = components or {}
            components[Worlds.ClientTag] = true
            return evolved.spawn(components)
        end,
    }
end

return Worlds
```

This approach respects evolved.lua's singleton design while maintaining clean server/client separation. Queries filter by tag to get "world-scoped" results.

```lua
local Context = {}
Context.__index = Context

function Context.new(opts)
    opts = opts or {}
    local services = Services.new()
    return setmetatable({
        worlds = opts.worlds,
        bus = opts.bus,
        config = opts.config or {},
        services = services,
    }, Context)
end
```

The **Services** sub-object (defined in the same file or as a local):
- `services:register(name, provider)` — stores a named service. Error if name already registered.
- `services:get(name)` — returns the service. Error with descriptive message if not found: `"Service 'X' not found — did the owning plugin forget to register it?"`

Per CONTEXT.md decisions:
- Services are **stateless queries** — no ECS mutations, no event emission internally. This is a convention enforced by the architecture validator (future extension), not by the Context code itself.
- Fail-fast on missing/undeclared services at boot.
